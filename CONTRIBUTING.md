# 贡献指南

这个项目可以接受快速迭代，但不能接受没有上下文、没有自测、没有审核记录的 PR。所有 PR 默认按下面流程处理。

## PR 基本要求

1. PR 必须使用默认模板，写清改动目的、影响范围、风险说明和自测记录。
2. 非维护性改动必须说明用户可见行为变化。涉及 UI 的改动需要截图或录屏。
3. 涉及 `manifest.json`、权限、后台脚本、数据库、发布流程、GitHub Actions、安全清理逻辑的改动，必须在 PR 描述中点名要求维护者重点审核。
4. 不接受把多个无关改动塞进同一个 PR。大改动应拆成可独立 review 的小 PR。
5. 不接受只写“fix bug”“update code”之类没有信息量的标题或描述。

## 审核规则

- 合并到 `main` 前至少需要 1 个维护者 approval。
- 作者不能自审自合。
- 高风险改动建议至少 2 个维护者看过，或由 code owner 明确确认。
- Review 不只看代码能不能跑，还要看是否破坏 7FA4 页面兼容性、是否引入 XSS/权限/数据泄露风险、是否影响发布包。
- 如果 PR 在 review 后继续 push，reviewer 需要重新确认新增 diff。

## 合并前检查

合并者需要确认：

- PR 描述通过 `PR Review Policy` 检查。
- 必要的手动测试已经写在 PR 里。
- 变更范围和 PR 标题一致。
- 没有调试日志、测试账号、Cookie、密钥或本地路径泄露。
- 对用户数据、数据库、权限和发布流程的影响已经明确。

## 建议的自测范围

按改动选择对应页面测试：

- 常规页面：题目列表、题目详情、个人计划、复盘本、比赛页。
- 面板：打开/关闭、保存配置、取消更改、主题、颜色、隐藏头像、更新提示。
- 聊天室：登录状态、会话列表、发送消息、预览、历史消息、新消息提示。
- 提交器：更新登录信息、外站提交记录发送、错误提示。
- 发布相关：`python build.py test`、`python build.py build`、Release 包目录结构。
